## 3. 本次构建发现的问题（已修，按出现顺序）

### 3.1 `CMakeLists.txt`：交叉编译时 host tag 用错

**症状**：在 Windows host 上交叉编译到 Android，但 cmake 把 NDK host 路径设成了 `linux-x86_64`，导致 tokenizers-cpp 的 Rust 子构建拿到了 Linux 路径的 clang。

**根因**：原 `CMakeLists.txt` 用 `if(WIN32)` 判断 host，但 CMake 交叉编译到 Android 时 `WIN32` 反映的是 **TARGET**（Android ≠ Windows），所以判断永远是 `false`。

**修复**：改用 `CMAKE_HOST_WIN32` / `CMAKE_HOST_APPLE`。

```cmake
# 旧
if(WIN32)
    set(HOST_TAG "windows-x86_64")
else()
    set(HOST_TAG "linux-x86_64")
endif()

# 新
if(CMAKE_HOST_WIN32)
    set(HOST_TAG "windows-x86_64")
elseif(CMAKE_HOST_APPLE)
    set(HOST_TAG "darwin-x86_64")
else()
    set(HOST_TAG "linux-x86_64")
endif()
```

⚠️ `ANDROID_TOOLCHAIN_ROOT` 是 `CACHE PATH`，第一次 configure 写错后即使改了源码，重新 configure 也不会刷新。要清掉 cache：
```
cmake --preset android-release -UANDROID_TOOLCHAIN_ROOT
```
或直接删 `build/android/CMakeCache.txt`。

### 3.2 QNN 2.45 API 变化：`extractBackendProfilingInfo` 多了一个参数

**症状**：
```
src/QnnModel.hpp:749:57: error: too few arguments to function call, expected 2, have 1
      extractBackendProfilingInfo(m_profileBackendHandle);
```

**根因**：QNN 2.45 SampleApp 把签名改成
```cpp
StatusCode extractBackendProfilingInfo(Qnn_ProfileHandle_t profileHandle,
                                       QnnSystemProfile_ProfileData_t *profileData);
```
我们的 `QnnModel.hpp:749` 仍用 1 参旧签名。

**修复**：传 `nullptr` 占位（与 2.45 SampleApp 自己在 `m_dumpOutputs=false` 时的写法一致）。
```cpp
extractBackendProfilingInfo(m_profileBackendHandle, nullptr);
```

> 这是目前**唯一**因 2.39→2.45 API 差异引起的源码修改。后续若打开 dump-profile 路径，可考虑透传一个真正的 `QnnSystemProfile_ProfileData_t` 上来。

### 3.3 tokenizers-cpp Rust 代码触发 1.96 新错误：`dangerous_implicit_autorefs`

**症状**：
```
error: implicit autoref creates a reference to the dereference of a raw pointer
   --> rust\src\lib.rs:218
        *out_len = (*handle).decode_str.len();
```
Rust 1.96 把这个 lint 默认升级为 error。

**修复**：显式取引用。
```rust
// before
*out_len = (*handle).decode_str.len();
// after
*out_len = (&(*handle).decode_str).len();
```
共 2 处（lib.rs:218, 251）。已在本仓库 submodule 工作树修好；上游 tokenizers-cpp 截至本次构建未跟上 Rust 1.96。

⚠️ 这是 submodule 内部改动，commit/push 时要进 submodule 单独提（或考虑提交一个 patch 文件 + 构建期 apply）。

### 3.4 xtl submodule 被升到 0.8.2 后丢了 `conjunction` / `static_if`

**症状**：
```
xtensor/xshape.hpp:404: error: no template named 'conjunction' in namespace 'xtl'
xtensor/xiterator.hpp:423: error: no member named 'static_if' in namespace 'xtl::mpl'
```
xtl 0.8.0 起就移除了这两个工具，但当前 xtensor 还在用它们。

**根因**：之前在 2.39 时期 pull 过 submodule，工作树 HEAD 偏到了 0.8.2 (`c52350e`)。仓库 `git ls-tree HEAD` 显示的 pinned 指针是 **`d11fb6b`（0.8.0/0.8.1 之间）**，该 commit 还保留了 `conjunction`/`static_if`。

**修复**：把 xtl 回滚到 pinned commit。
```bash
cd app/src/main/cpp/3rdparty/xtl
git fetch origin --tags
git checkout d11fb6b5f4c417025124ed2c62175284846a1914
```
或者更简单：
```bash
git submodule update --init --recursive --force app/src/main/cpp/3rdparty/xtl
```

⚠️ 同理需要 audit 一下其它 submodule 是否也被 pull 偏了：MNN / cpp-httplib / json / xsimd / tokenizers-cpp / xtensor。本次构建只 xtl 卡了。

### 3.5 Rust toolchain 状态损坏

**症状**：rustup 一直说 `rust-std aarch64-linux-android is up to date`，但磁盘上 `lib/rustlib/aarch64-linux-android/` 目录根本不存在。直接 `cargo build` 报：
```
error[E0463]: can't find crate for `core`
note: the `aarch64-linux-android` target may not be installed
```

**根因**：曾经有一次安装中断 + 手动清理，导致：
- `lib/rustlib/aarch64-linux-android/` 目录被清空
- `lib/rustlib/manifest-rust-std-aarch64-linux-android` 也被清掉
- 但 `lib/rustlib/multirust-config.toml` 和 `lib/rustlib/components` 仍记录该 target「已安装」
- rustup 看到 toml 里有记录就跳过下载

**修复**：手动从两个状态文件里删掉 `rust-std aarch64-linux-android` 条目，再重装。
```bash
# 1. 从 components 文件移除
sed -i '/^rust-std-aarch64-linux-android$/d' \
  "D:/dev/rust/toolchains/stable-x86_64-pc-windows-msvc/lib/rustlib/components"

# 2. 从 multirust-config.toml 移除对应 [[components]] 块（4 行 + 空行）
#    用 python/手工编辑，删除：
#    [[components]]
#    pkg = "rust-std"
#    target = "aarch64-linux-android"
#    is_extension = false

# 3. 重新安装
rustup target add aarch64-linux-android
```

## 4. 下次构建快速路径

```bash
# === 一次性环境 ===
export QAIRT_PATH="D:/dev/qairt/2.45.41.260507"
export ANDROID_NDK_ROOT="D:/dev/android_sdk/ndk/28.2.13676358"
export PATH="/d/dev/android_sdk/cmake/3.22.1/bin:$PATH"

# === pre-check（理想情况都已 OK，跳过即可）===
rustup target list --installed | grep -q aarch64-linux-android || rustup target add aarch64-linux-android
ls "$ANDROID_NDK_ROOT/toolchains/llvm/prebuilt/windows-x86_64/bin/aarch64-linux-android21-clang.cmd" >/dev/null

# === build ===
cd app/src/main/cpp
cmake.exe --preset android-release
cmake.exe --build --preset android-release

# === 拷贝产物 ===
cp build/android/bin/arm64-v8a/libstable_diffusion_core.so ../jniLibs/arm64-v8a/
cp build/android/qnnlibs/*.so ../assets/qnnlibs/
```

或直接跑 `build.bat`（在 Developer Command Prompt for VS / 已把 cmake+ninja 加进 PATH 的环境里）。

## 6. 构建脚本改进建议

老 `build.bat` 只是顺序跑 cmake，没有处理：

1. 环境变量校验（QAIRT_PATH / ANDROID_NDK_ROOT / cmake/ninja in PATH）
2. Rust target 安装检查
3. submodule 偏移检测（`git submodule status` 里 `+` 前缀提示）
4. host tag cache 卡死时的 `-UANDROID_TOOLCHAIN_ROOT`

下一节落地一个 `build.ps1` 把这些 sanity check 都放进去。
